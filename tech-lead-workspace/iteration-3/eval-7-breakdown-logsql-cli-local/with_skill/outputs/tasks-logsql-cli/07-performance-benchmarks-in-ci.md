### Performance benchmarks in CI: cold-start and throughput

**One-liner:** Wire the two NFR-stated targets — first-row < 100ms and 1 GB / 60s throughput — into CI as hard-fail benchmarks so a regression doesn't ship to engineers' laptops unnoticed.

**Composes:**
- Cold-start benchmark: measure wall-clock time from process invocation to the first row of CSV output appearing on STDOUT for `SELECT * FROM stdin WHERE level = "error" LIMIT 10` against a fixed input fixture. Target: < 100 ms on the CI runner reference machine.
- Throughput benchmark: measure time to complete `SELECT count, msg FROM stdin WHERE status >= 500 LIMIT 1000000` against a generated 1 GB JSON-lines fixture. Target: < 60 s on the CI runner reference machine.
- Both benchmarks have explicit numeric thresholds in the CI job; exceeding them fails the build, not just emits a warning.
- Synthetic 1 GB fixture generated deterministically in the benchmark job (not checked into the repo) so the throughput run is reproducible.
- Reference machine specs (CI runner type, CPU, RAM) documented next to the thresholds so future tightening or relaxing is intentional rather than tuned to whichever runner happened to be available.
- Benchmark runs are isolated from coverage / race-detector builds so timing is representative.

**TDD sections addressed:** §6 NFR: cold-start latency (verified in CI), §6 NFR: throughput (verified in CI).

**Depends on:** Walking skeleton: SELECT / WHERE / LIMIT over JSON-lines to CSV, JSON-lines output mode, GROUP BY with `COUNT`, `SUM`, `AVG`, `MIN`, `MAX`, `COUNT(DISTINCT)` under a memory budget, ORDER BY operator under the memory budget

**Acceptance criteria:**
- CI pipeline has a benchmark job that runs on every PR to the release branch; the job fails the build if cold-start exceeds 100 ms or throughput exceeds 60 s for the 1 GB fixture.
- The cold-start benchmark measures from process spawn to first byte of CSV on STDOUT — not from `main()` entry — so it captures any Go runtime initialization overhead engineers will see.
- A deliberate regression (e.g. inserting a 200 ms sleep at startup) fails the CI benchmark job.
- The 1 GB synthetic fixture is generated by the job from a documented seed, not checked into the repo, and the generation step itself is excluded from the throughput timing.
- Benchmark thresholds, reference runner type, and fixture-generation method are documented in the repo so future changes are reviewed deliberately.
