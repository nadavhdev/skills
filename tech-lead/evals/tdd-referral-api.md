# TDD: Referral codes — API

**Author:** Tech lead (drafted on behalf of the product team)
**Status:** Draft
**Date:** 2026-06-08
**Related PRD:** `evals/prd-referral-api.md`

## 1. Problem & context

Mobile users can generate a short referral code, share it, and a new user
who redeems that code grants both parties a $10 credit on their Stripe
balance. The Q3 launch driver is an acquisition push targeted at iOS in
the US. Mobile clients call this service directly; the web app calls it
through the existing BFF.

## 2. Scope

**In scope:** code issuance, code redemption (one-shot), credit posting to
the Stripe ledger, a dashboard read path for support to look up a code
and its redemption status.

**Out of scope:** fraud / abuse policy (see Open Questions §10.1),
expiry-policy enforcement beyond a hard 90-day TTL, multi-currency
credits, payout-style cash referrals, and any change to the Stripe
ledger schema itself.

## 3. High-level approach

A new synchronous JSON-over-HTTPS service `referrals-api` exposes three
endpoints (`POST /codes`, `POST /codes/{code}/redeem`, `GET /codes/{code}`),
sitting behind the existing API gateway. State lives in two Postgres
tables (`referral_codes`, `referral_redemptions`) on the existing
`growth` cluster. Redemption is the only call path that mutates the
credit ledger, and it does so via the existing internal
`credit-ledger-api` (a separately-owned service we call as a client).

Redemption is the load-bearing operation: every other endpoint reads, and
issuance is cheap. The redemption path must be idempotent on a
client-supplied `Idempotency-Key` header so that mobile retries (which
will happen on flaky networks) don't double-credit.

## 4. Detailed design

### 4.1 Endpoints

- `POST /codes` — issues a code for the authenticated user. Returns
  `{code, expires_at}`. One active code per user at a time; re-issuing
  invalidates the previous code.
- `POST /codes/{code}/redeem` — called by the *referee* (the new user).
  Body carries the referee's user id (derived from the auth token).
  Requires `Idempotency-Key` header. On success, both the referrer and
  the referee receive a $10 credit via `credit-ledger-api`.
- `GET /codes/{code}` — used by the support dashboard. Returns the code,
  issuer, status, expiry, and redemption record if any. Auth-scoped to
  support role.

### 4.2 Data model (Postgres)

`referral_codes(code PK, issuer_user_id, status, created_at, expires_at)`
with a partial unique index `(issuer_user_id) WHERE status='active'` to
enforce one active code per user.

`referral_redemptions(id PK, code FK, referee_user_id, idempotency_key,
status, credited_at)` with unique constraint on `(code)` (one redemption
per code) and `(idempotency_key, code)` (idempotency).

### 4.3 Code generation

8-character base32 (Crockford), generated by the API itself with a
collision retry loop bounded at 5 attempts. At 10⁵ codes the expected
collision rate is < 10⁻⁵, so the retry is a backstop, not a hot path.

### 4.4 Credit posting

Redemption opens a Postgres transaction, inserts the redemption row,
and calls `credit-ledger-api` inside the transaction. If the credit
call fails, the transaction rolls back. The credit-ledger call carries
the same `Idempotency-Key` so retries by us are safe on its side.

### 4.5 AuthN/Z

Mobile and web clients pass an existing user-scoped JWT. Support
clients pass a separate ops JWT with a `support` claim required for
`GET /codes/{code}`.

## 5. Alternatives considered

- **Issue codes ahead of time and pre-load a pool.** Rejected: makes
  the dashboard read path more complex and gives no measurable win for
  our QPS.
- **Embed referee credit in the JWT, redeem at first purchase.**
  Rejected: couples referrals to the checkout team's deploy schedule.
- **Do nothing — keep the existing manual code allowlist.** Rejected:
  product won't ship Q3 without a self-serve path.

## 6. Non-functional requirements

- **Latency.** p99 < 250ms for redemption; p99 < 150ms for issuance and
  read. *How met:* synchronous Postgres + one credit-ledger call;
  ledger call has its own p99 SLO of 100ms. *Verified:* load test at
  3× peak.
- **Throughput.** 50 redemptions/sec sustained, 200/sec peak (launch +
  flash promo). *How met:* single-region two-AZ deploy, 4 pods.
- **Availability.** 99.9% monthly. *How met:* standard service tier.
- **Durability.** Postgres point-in-time recovery, 7-day window.
- **Security.** PII (user id) only; no card data. JWT validation at the
  gateway, role checks at the handler.
- **Cost.** N/A — fits in existing growth-cluster capacity.
- **Compliance.** N/A — no new regulated data classes.
- **DR.** N/A — single-region acceptable for this launch (degrades
  to "cannot redeem", does not corrupt state).

## 7. Risks & failure modes

- **Credit-ledger outage.** Redemptions fail loud. Client retries with
  same idempotency key once the ledger recovers. No silent retry loop
  in our service.
- **Concurrent redemption of the same code.** Unique constraint on
  `referral_redemptions(code)` produces a `unique_violation` on the
  loser; we map to `409 Conflict` with the existing redemption details.
- **Mobile client double-tap.** Caught by `Idempotency-Key`.
- **Promo-driven traffic spike.** Throughput SLO above accounts for
  3× peak; alert on saturation > 70%.

## 8. Rollout & migration

- Deploy behind a launch flag `referrals_api_enabled`, default off.
- Internal QA flips it on for staff users only for one week.
- Gradual: 1% → 10% → 100% of US iOS traffic over 5 days. Web behind
  separate flag flip.
- Backwards compat: N/A — new service.
- Rollback: flip flag off; in-flight redemptions complete (synchronous).

## 9. Observability & operations

- Metrics: `referral_issuance_total`, `referral_redemption_total{result}`,
  `credit_ledger_latency_seconds`, gateway request rate / error rate.
- Logs: structured JSON, include `code`, `idempotency_key`,
  `referee_user_id`, never the JWT.
- Alerts: redemption error rate > 1% for 5 min; credit-ledger call p99
  > 250ms for 10 min; saturation > 70% for 10 min.
- Runbook entries: ledger-outage drill, idempotency-key collision
  diagnosis, code-collision rate alarm investigation.

## 10. Open questions

1. **Fraud / self-referral / multi-account abuse policy.** Product owes
   us a decision on whether a referee on the same device, same payment
   instrument, or same household as the referrer counts as a valid
   redemption. Until decided, we cannot scope the abuse-check task.
2. **Code expiry behavior at the boundary.** When a code expires *while*
   a redemption is in flight (request received pre-expiry, processed
   post-expiry), do we honor it? Recommend: honor if the request
   arrived before `expires_at`; need product sign-off.
